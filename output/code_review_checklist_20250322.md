## 一、代码质量检查

### 1.1 可读性与命名规范

| 检查点 | 说明 | 优先级 |
|--------|------|--------|
| 命名具有描述性 | 变量、函数、类名应清晰表达其用途，避免模糊缩写（如 `a`, `tmp`, `data1`） | 高 |
| 遵循命名约定 | 统一使用驼峰式(camelCase)、蛇形式(snake_case)或帕斯卡式(PascalCase)，与项目规范一致 | 高 |
| 常量全大写 | 常量使用全大写加下划线（如 `MAX_RETRY_COUNT`） | 中 |
| 布尔变量语义清晰 | 布尔变量名应能回答 true/false 问题（如 `isValid`, `hasPermission`） | 中 |
| 避免魔法数字/字符串 | 将硬编码值提取为命名常量 | 中 |
| 代码缩进一致 | 使用统一的缩进风格（空格或Tab），通常为2或4个空格 | 高 |
| 行长度限制 | 单行不超过80-120字符，超长应换行 | 低 |

### 1.2 代码复杂度与函数长度

| 检查点 | 说明 | 优先级 |
|--------|------|--------|
| 函数单一职责 | 每个函数只做一件事，功能边界清晰 | 高 |
| 函数长度控制 | 函数行数不超过50行，理想情况下20行以内 | 中 |
| 参数数量控制 | 函数参数不超过4-5个，过多应使用对象封装 | 中 |
| 圈复杂度控制 | 避免深层嵌套（if/for/while），圈复杂度建议≤10 | 中 |
| 尽早返回 | 使用卫语句(guard clause)减少嵌套层级 | 中 |
| 避免过长代码块 | 代码块（if/for/while 体内）不超过15-20行 | 低 |
| 分解复杂表达式 | 复杂条件判断应提取为变量或函数 | 中 |

### 1.3 注释与文档

| 检查点 | 说明 | 优先级 |
|--------|------|--------|
| 公共API文档 | 所有公共函数、类、接口必须有文档注释（JSDoc/Docstring等） | 高 |
| 复杂逻辑注释 | 非显而易见的算法或业务逻辑需要解释说明 | 高 |
| TODO/FIXME标记 | 临时代码需标记 TODO 或 FIXME 并关联任务ID | 中 |
| 避免冗余注释 | 代码本身清晰时不应添加无意义的注释 | 中 |
| 注释与代码同步 | 修改代码时必须同步更新相关注释 | 高 |
| 文件头注释 | 关键文件应包含作者、创建日期、功能描述 | 低 |
| 注释格式规范 | 使用统一的注释格式和风格 | 低 |

## 附录

### 优先级说明

- **高**: 必须检查项，违反可能导致严重问题
- **中**: 推荐检查项，影响代码质量和维护性
- **低**: 建议检查项，提升代码优雅度

### 使用建议

1. 根据项目实际情况调整检查项的优先级
2. 可结合自动化工具（ESLint/SonarQube/CodeQL）进行初步筛选
3. 定期回顾和更新本检查清单
4. 新人入职培训时作为参考材料
